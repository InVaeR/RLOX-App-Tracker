Привет! Для такого перехода лучше разделить продукт на три независимых компонента:

1. **RLOX App Tracker** — основное Python/PySide6-приложение.
2. **RLOX Launcher** — стабильная точка входа, которая проверяет обновления и запускает приложение.
3. **RLOX App Tracker Setup** — установщик, который устанавливает/обновляет файлы, ярлыки, автозапуск и запись удаления.

Не рекомендую давать лаунчеру самостоятельно перезаписывать запущенные EXE-файлы. Надёжнее, если он загружает новый установщик и запускает его в режиме обновления. Так один механизм будет отвечать и за первую установку, и за последующие обновления.

# 1. Целевая архитектура

## Компоненты

### Основное приложение

Остаётся на Python/PySide6 и содержит:

- трекер;
- SQLite;
- интерфейс;
- трей;
- настройки;
- экспорт;
- миграции базы данных.

Из него удаляется непосредственная загрузка обновлений. Приложение только вызывает лаунчер.

### Лаунчер

Лаунчер должен:

- определять, установлено ли приложение;
- читать установленную версию;
- проверять манифест обновлений;
- скачивать установщик;
- проверять размер, SHA-256 и цифровую подпись;
- корректно завершать приложение перед обновлением;
- запускать установщик;
- запускать текущую версию приложения;
- поддерживать интерактивный и фоновый режим;
- показывать ошибки;
- иметь собственный лог;
- при необходимости управлять автозапуском.

### Установщик

Установщик должен:

- поддерживать первую установку;
- поддерживать обновление поверх существующей установки;
- регистрировать программу в списке установленных приложений Windows;
- создавать ярлыки;
- устанавливать лаунчер и приложение;
- создавать деинсталлятор;
- опционально включать автозапуск;
- не удалять пользовательскую БД при обновлении;
- по умолчанию не удалять данные при деинсталляции;
- мигрировать данные старого RusLOXPy.

Для Windows удобно использовать **Inno Setup**.

# 2. Рекомендуемый технологический стек

## Основное приложение

Оставить:

- Python 3.11 или перейти на Python 3.12 после тестирования;
- PySide6;
- psutil;
- pywin32;
- SQLite;
- PyInstaller в режиме `onedir`.

## Лаунчер

Рекомендую:

- C#;
- .NET 8;
- WinForms;
- публикация `win-x64`, self-contained;
- single-file, если не возникнет проблем с Native DLL.

Почему не стоит писать лаунчер на PySide6:

- второй комплект Qt значительно увеличит размер;
- лаунчер будет зависеть от тех же библиотек, которые он должен обновлять;
- обновление самого лаунчера станет сложнее;
- запуск будет медленнее.

Допустимый MVP-вариант — отдельный Python-лаунчер на стандартной библиотеке, собранный PyInstaller. Но C# для системного Windows-компонента будет надёжнее.

## Установщик

- Inno Setup 6;
- per-user установка без прав администратора;
- при необходимости позже добавить machine-wide установку.

# 4. Расположение файлов в системе

## Каталог установки

Для установки без прав администратора:

```text
%LOCALAPPDATA%\Programs\RLOX App Tracker\
├── RLOXLauncher.exe
├── launcher.version
├── versions\
│   ├── 2.0.0\
│   │   ├── RLOXAppTracker.exe
│   │   ├── assets\
│   │   ├── PySide6\
│   │   └── ...
│   └── 2.0.1\
│       └── ...
├── state\
│   └── install.json
└── unins000.exe
```

Не рекомендуется устанавливать Python-приложение в один EXE. Для установленного продукта лучше `onedir`:

- приложение запускается быстрее;
- нет распаковки во временную папку;
- установщик и так сжимает файлы;
- проще диагностировать проблемы;
- проще хранить несколько версий;
- проще откатывать обновления.

## Пользовательские данные

Рекомендуемый каталог:

```text
%LOCALAPPDATA%\RLOX App Tracker\
├── data\
│   └── tracker.db
├── config\
│   ├── app.json
│   └── launcher.json
├── logs\
│   ├── app.log
│   └── launcher.log
├── updates\
│   └── downloads\
└── migration\
    └── ...
```

БД и настройки никогда не должны находиться внутри каталога установки или каталога версии.

## Почему `LOCALAPPDATA`

SQLite-база постоянно изменяется и обычно не должна синхронизироваться через roaming profile. Поэтому для БД предпочтительнее `%LOCALAPPDATA%`, а не `%APPDATA%`.

# 6. Как переносить существующий проект

## Шаг 1. Создать новый репозиторий с историей

Предпочтительно сохранить Git-историю:

```bash
git clone https://github.com/InVaeR/RusLOXPy.git RLOX-App-Tracker
cd RLOX-App-Tracker
git remote rename origin old-origin
git remote add origin https://github.com/InVaeR/RLOX-App-Tracker.git
git checkout -b main
git push -u origin main
```

После этого:

- старый репозиторий оставить доступным;
- в его README добавить сообщение о новом проекте;
- не удалять старые релизы;
- в новом репозитории сохранить MIT License;
- отметить, что проект основан на RusLOXPy.

## Шаг 2. Сначала перенести приложение без изменения логики

На первом этапе нельзя одновременно:

- переименовывать всё;
- менять структуру;
- менять обновление;
- менять БД;
- менять сборку.

Сначала нужно получить рабочее приложение в новом репозитории.

Перенос:

```text
core/       -> app/src/rlox_app_tracker/core/
data/       -> app/src/rlox_app_tracker/data/
services/   -> app/src/rlox_app_tracker/services/
ui/         -> app/src/rlox_app_tracker/ui/
utils/      -> app/src/rlox_app_tracker/utils/
main.py     -> app/src/rlox_app_tracker/__main__.py
config.py   -> app/src/rlox_app_tracker/paths.py + settings.py
assets/     -> app/assets/
```

После переноса заменить абсолютные импорты вроде:

```python
from config import DATA_DIR
```

на пакетные:

```python
from rlox_app_tracker.paths import DATA_DIR
```

Это устранит конфликты модулей и упростит тестирование.

## Шаг 3. Разделить конфигурацию и пути

Текущий `config.py` смешивает:

- название;
- версию;
- системные пути;
- значения настроек;
- путь к ресурсам.

Нужно разделить:

```text
version.py       — версия и build metadata
paths.py         — install/data/log/config paths
defaults.py      — значения настроек по умолчанию
metadata.py      — ProductName, Publisher, AppId
```

# 7. Изменения в основном приложении

## 7.1. Переименование

Заменить:

- `RusLOXPy` → `RLOX App Tracker`;
- `RusLOXPy.exe` → `RLOXAppTracker.exe`;
- старые URL репозитория;
- подписи окон;
- tray tooltip;
- экспортируемые имена файлов;
- singleton key;
- данные version resource;
- User-Agent.

При этом старое имя `RusLOXPy` оставить только в модуле миграции.

## 7.2. Исправить систему версий

Текущая версия формируется через текущую дату:

```python
__date_stamp__ = date.today().strftime("%Y%m%d")
```

Для установленного приложения это нежелательно. Один и тот же исходный код может показывать разную версию в зависимости от даты запуска или сборки.

Использовать SemVer:

```text
2.0.0
2.0.1
2.1.0
3.0.0
```

Версия должна поступать:

- из Git-тега;
- либо из сгенерированного файла во время CI;
- либо из `pyproject.toml`.

Дополнительную информацию о сборке можно хранить отдельно:

```text
Version: 2.0.0
Commit: a1b2c3d
Build date: 2026-07-13
Channel: stable
```

Тег релиза:

```text
v2.0.0
```

## 7.3. Удалить старую логику обновления

Сейчас `services/updater.py` только проверяет GitHub Release и открывает браузер. После введения лаунчера:

- удалить загрузку GitHub API из основного приложения;
- удалить `services/update_worker.py`;
- заменить `services/updater.py` на `launcher_bridge.py`;
- кнопка «Проверить обновления» должна запускать лаунчер.

Пример командного контракта:

```text
RLOXLauncher.exe --check-updates --interactive
RLOXLauncher.exe --check-updates --silent
RLOXLauncher.exe --launch
RLOXLauncher.exe --launch --background
RLOXLauncher.exe --repair
```

Основное приложение не должно знать URL установщика и алгоритм обновления.

## 7.4. Изменить автозапуск

Сейчас автозапуск записывает `sys.executable`, то есть конкретный EXE приложения. После обновления путь может измениться.

В автозапуске всегда должен быть стабильный лаунчер:

```text
"%LOCALAPPDATA%\Programs\RLOX App Tracker\RLOXLauncher.exe" --launch --background
```

Запись:

```text
HKCU\Software\Microsoft\Windows\CurrentVersion\Run
RLOXAppTracker
```

Управлять этой записью может:

- установщик при первой установке;
- лаунчер при изменении настройки;
- приложение через вызов лаунчера.

Основному приложению больше не следует самостоятельно писать путь `sys.executable` в реестр.

## 7.5. Поддержать аргументы запуска

Основное приложение должно понимать:

```text
--background
--minimized
--after-update
--data-dir PATH
--version
```

Поведение `--background`:

- не показывать главное окно;
- сразу создать tray icon;
- запустить трекер;
- не показывать окно обновления.

## 7.6. Улучшить single-instance

Сейчас второй экземпляр просто показывает сообщение и завершается. Лучше:

- первый экземпляр создаёт `QLocalServer`;
- второй отправляет команду `show`;
- первый показывает и активирует существующее окно;
- лаунчер может отправить `shutdown-for-update`;
- приложение закрывает сессии и БД;
- затем подтверждает завершение.

Команды IPC:

```text
show
shutdown
shutdown-for-update
pause
resume
ping
```

Если IPC недоступен, лаунчер может ждать обычного завершения процесса, а принудительное завершение использовать только после подтверждения пользователя.

## 7.7. Корректное завершение

Нужно проверить, что:

- `tracker.stop()` вызывается один раз;
- активные сессии сохраняются;
- SQLite WAL корректно закрывается;
- `db.close()` вызывается после трекера;
- обновление не начинается, пока БД открыта процессом.

Сейчас остановка вызывается из `_quit()` и затем ещё раз через `aboutToQuit`. Лучше сделать очистку идемпотентной.

## 7.8. Пути ресурсов

Для `onedir`-сборки путь приложения нужно определять явно:

- при разработке — относительно исходников;
- в frozen-режиме — относительно каталога EXE;
- пользовательские данные — только в `%LOCALAPPDATA%`.

Не следует считать, что `Path(__file__).parent` всегда является каталогом установленного приложения.

# 8. Миграция данных RusLOXPy

Старые данные находятся здесь:

```text
%APPDATA%\RusLOXPy\
├── tracker.db
├── tracker.db-wal
├── tracker.db-shm
├── config.json
└── app.log
```

Новые:

```text
%LOCALAPPDATA%\RLOX App Tracker\
```

## Алгоритм миграции

При первом запуске новой версии:

1. Проверить наличие маркера завершённой миграции.
2. Проверить наличие новых данных.
3. Проверить `%APPDATA%\RusLOXPy`.
4. Если новая БД отсутствует:
   - убедиться, что старое приложение не запущено;
   - скопировать `tracker.db`;
   - при наличии скопировать `-wal` и `-shm`;
   - лучше перед копированием выполнить SQLite checkpoint через отдельное подключение;
   - создать резервную копию;
   - скопировать конфигурацию;
   - преобразовать ключи конфигурации;
   - записать migration report.
5. Не удалять старые данные автоматически.
6. Создать маркер успешной миграции.

Пример:

```text
%LOCALAPPDATA%\RLOX App Tracker\migration\rusloxpy-1.x.json
```

Он должен содержать:

- дату;
- исходный каталог;
- скопированные файлы;
- результат;
- исходную версию;
- целевую версию.

## Старый автозапуск

Нужно проверить значение `RusLOXPy` в `HKCU\...\Run`.

Удалять его можно только если:

- значение действительно указывает на старый `RusLOXPy.exe`;
- пользователь согласился на перенос;
- новая запись автозапуска успешно создана.

Не следует удалять старый EXE из папки загрузок пользователя.

# 9. Протокол обновления

## Манифест

Лаунчер не должен искать случайный EXE в списке assets. Релиз должен содержать строгий манифест.

Пример `latest.json`:

```json
{
  "schemaVersion": 1,
  "product": "rlox-app-tracker",
  "channel": "stable",
  "version": "2.0.1",
  "minimumLauncherVersion": "1.0.0",
  "publishedAt": "2026-07-13T12:00:00Z",
  "mandatory": false,
  "installer": {
    "url": "https://github.com/InVaeR/RLOX-App-Tracker/releases/download/v2.0.1/RLOX-App-Tracker-Setup-2.0.1-x64.exe",
    "sha256": "HEX_SHA256_VALUE",
    "size": 123456789
  },
  "releaseNotesUrl": "https://github.com/InVaeR/RLOX-App-Tracker/releases/tag/v2.0.1"
}
```

Стабильный URL:

```text
https://github.com/InVaeR/RLOX-App-Tracker/releases/latest/download/latest.json
```

## Состояние установленной версии

Пример `install.json`:

```json
{
  "schemaVersion": 1,
  "currentVersion": "2.0.1",
  "previousVersion": "2.0.0",
  "channel": "stable",
  "installedAt": "2026-07-13T12:15:00Z",
  "appExecutable": "versions\\2.0.1\\RLOXAppTracker.exe"
}
```

Записывать файл нужно атомарно:

1. создать `install.json.tmp`;
2. записать и закрыть;
3. заменить основной файл через rename/replace.

# 10. Алгоритм работы лаунчера

## Обычный запуск

1. Запустить singleton лаунчера.
2. Прочитать конфигурацию.
3. Проверить `install.json`.
4. Если приложение уже запущено:
   - отправить IPC-команду `show`;
   - завершить лаунчер.
5. Определить, нужно ли проверять обновления.
6. Если проверка не требуется — запустить приложение.
7. Если требуется:
   - скачать `latest.json`;
   - проверить формат;
   - сравнить SemVer;
   - при отсутствии обновления запустить приложение;
   - при ошибке сети запустить установленную версию.
8. Если обновление доступно:
   - применить выбранную политику;
   - скачать установщик;
   - проверить его;
   - запустить обновление.

Отсутствие интернета не должно блокировать приложение.

## Фоновый автозапуск

Команда:

```text
RLOXLauncher.exe --launch --background
```

Правила:

- не показывать главное окно лаунчера без необходимости;
- ограничить сетевую проверку коротким таймаутом;
- при ошибке сразу запускать установленную версию;
- необязательное обновление не должно мешать загрузке Windows;
- обязательное обновление можно установить до запуска приложения.

## Ручная проверка

Команда:

```text
RLOXLauncher.exe --check-updates --interactive
```

Лаунчер показывает:

- текущую версию;
- новую версию;
- release notes;
- размер;

## Установка обновления


2. После завершения переименовать в:

```text
setup.exe
```

# 12. Версионные каталоги и откат

Лучше хранить приложение по версиям:

```text
versions\
├── 2.0.0\
└── 2.0.1\
```

После успешного обновления:

- `currentVersion = 2.0.1`;
- `previousVersion = 2.0.0`;
- хранить максимум две последние версии;
- более старые удалять после подтверждённого запуска.

## Проверка успешного запуска

Новое приложение после инициализации должно создавать marker:

```text
state\startup-ok-2.0.1
```

Создавать marker только после того, как:

- открыта БД;
- применены миграции;
- создано главное окно;
- запущен tracker;
- создан tray icon.

Если приложение аварийно завершается до marker:

- лаунчер фиксирует failed startup;
- не удаляет предыдущую версию;
- предлагает откат;
- переключает `currentVersion` обратно.

Важно: откат приложения не всегда означает откат БД. Поэтому миграции БД должны быть обратно совместимыми хотя бы на одну версию либо перед опасной миграцией должна создаваться резервная копия.

# 15. Безопасность обновлений

Одного HTTPS недостаточно.

## Минимум для первой версии

- скачивание только по HTTPS;
- строгий JSON schema;
- проверка SHA-256 установщика;
- проверка ожидаемого имени продукта;
- запрет downgrade без специального флага;
- защита от path traversal;
- ограничение размера загрузки;
- таймауты;
- загрузка через временный `.part`;
- проверка exit code установщика.

## Для production

- подписывать Setup и EXE через Authenticode;
- проверять подпись через WinVerifyTrust;
- проверять издателя;
- подписывать `latest.json` отдельным ключом;
- публичный ключ манифеста встраивать в launcher;
- приватный ключ хранить только в GitHub Secrets или защищённом signing service.

SHA-256 из неподписанного манифеста защищает от повреждения файла, но не полностью защищает от компрометации источника манифеста.

# 23. Итоговая рекомендация

Оптимальная схема для этого проекта:

```text
GitHub Release
      │
      ├── latest.json
      └── RLOX-App-Tracker-Setup-x64.exe
                   │
                   ▼
          RLOXLauncher.exe
          проверка/скачивание
                   │
                   ▼
            Inno Setup
        установка/обновление
                   │
                   ▼
 versions\<version>\RLOXAppTracker.exe
                   │

```

Главные архитектурные правила:

- пользователь всегда запускает **launcher**;
- ярлыки и автозапуск всегда указывают на **launcher**;
- launcher никогда не хранится внутри каталога конкретной версии;
- приложение не обновляет себя самостоятельно;
- обновление выполняет Setup;
- БД не находится в каталоге установки;
- версии приложения устанавливаются раздельно;
- без интернета всегда запускается последняя рабочая версия;
- обновление проверяется по манифесту и обязательно валидируется;
- старая БД RusLOXPy мигрируется один раз и не удаляется автоматически.

Такой подход даст нормальную установку Windows, безопасные обновления, стабильный автозапуск и возможность дальнейшего отката без полной переработки архитектуры.
